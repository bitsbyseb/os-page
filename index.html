R1-S1 Evolución Histórica: De UNIX a Windows 11

Lo más importante que me quedó de este texto es cómo pasamos de sistemas gigantes que solo hacían una cosa a la vez, a lo que usamos hoy.

  • UNIX (1969): Es la base de todo. Ken Thompson y Dennis Ritchie lo crearon con una filosofía de "herramientas pequeñas" y diseño multitarea/multiusuario. Al escribirlo en C, se volvió portable, algo que cambió las reglas del juego.
  • Linux (1991): Linus Torvalds lanzó el kernel y, al ser código abierto (GPL), permitió que todo el mundo colaborara. Hoy domina la nube, servidores y es la base de Android.
  • macOS: Me pareció interesante que Apple usa una base UNIX llamada Darwin (mezcla de BSD y el microkernel Mach) para ser estable y seguro.
  • Windows: Pasó de ser una interfaz sobre MS-DOS a ser el estándar en computación personal, llegando a Windows 11 que ya integra hasta aplicaciones de Android y un entorno híbrido con Linux mediante WSL 2 (justo lo que estoy usando).

🐧 Mi experiencia: Instalando Arch Linux en WSL

Instalar Arch en WSL no fue como bajar una App de la Store. Fue una experiencia de instalación manual, muy similar a lo que harías en una máquina física, pero aprovechando la línea de comandos de Windows.

1. El despliegue inicial: Lo hice de la forma más directa: usando la herramienta de línea de comandos de WSL. Descargué el tarball (el archivo raíz) de Arch y lo importé directamente con el comando wsl --import. Fue mucho más sencillo y limpio que usar instaladores de terceros; en un par de segundos ya tenía el contenedor listo para arrancar.

2. Un sistema "desnudo" (Root): Al entrar por primera vez (wsl -d Arch), me di cuenta de que estaba totalmente solo. No había usuarios, no había seguridad, solo era yo como root. Arch viene minimalista de verdad: Contraseñas: Lo primero fue asignarle una clave al root con passwd, porque venía abierto. Sin usuario personal: No existía un "Johan". Tuve que crearlo manualmente con useradd -m -G wheel johan.

3. Construyendo el entorno (Paquetes básicos): Como buen Arch, no traía nada. Tuve que configurar los mirrors de pacman y empezar a instalar mis herramientas esenciales:

  • Sudo: ¡Fundamental! Al principio no podía hacer nada como usuario normal. Instalé sudo, edité el archivo visudo y me di permisos para poder administrar el sistema sin ser root todo el tiempo.
  • Editores: Para un fan de la terminal como yo, esto era prioridad. Instalé vi, vim y, por supuesto, neovim (nvim) para tener una configuración de desarrollo decente.
  • Base-devel: Instalé el grupo de paquetes básicos para poder compilar y tener las herramientas esenciales de Linux que no vienen por defecto.


4. Reflexión final: Aunque la instalación desde la CLI de Windows fue rápida, configurar el interior del sistema fue donde realmente aprendí. Tener que crear mi propio usuario, configurar los permisos de sudo y elegir cada editor de texto manualmente me recordó por qué Arch es la mejor forma de entender cómo funciona un sistema operativo por dentro. ¡Ahora tengo un entorno de desarrollo ultra ligero y a mi medida!

Planificación de Procesos: ¿Quién usa la CPU?

Aquí analizamos cómo el SO decide qué proceso se ejecuta. No hay un algoritmo perfecto; todo depende de qué necesitemos.

FIFO (First In, First Out): Es el más simple; el primero que llega es el primero que sale. Es justo, pero si llega un proceso muy largo al principio, bloquea a todos los demás (el famoso "Head-of-Line Blocking").

Round Robin (RR): Es el que usamos para sistemas interactivos. A cada proceso le damos un "pedacito" de tiempo (quantum). Si no termina, vuelve a la fila. Es muy equitativo.

SJF (Shortest Job First): Aquí el SO elige el proceso más corto para ejecutarlo primero. Minimiza el tiempo de espera promedio, pero el problema es que es difícil saber cuánto va a durar un proceso antes de que empiece.

CFS (Completely Fair Scheduler): Es el que usa Linux actualmente. En lugar de tiempos fijos, usa el vruntime (tiempo de ejecución virtual) para darle prioridad al proceso que menos tiempo ha usado en la CPU. Usa un "Árbol Rojo-Negro" para que todo sea súper rápido (eficiencia O(log N)).

Jerarquía de Memoria: Velocidad vs. Costo

Esto es clave para entender por qué mi PC no es más rápida de lo que ya es. Básicamente, organizamos la memoria en niveles para equilibrar velocidad y presupuesto.

  • Registros (L0): Están pegados a la CPU. Son los más rápidos (1 ciclo) pero apenas guardan unos pocos bytes.
  • Caché (L1, L2, L3): Memoria de alta velocidad para que la CPU no tenga que esperar tanto. L1 es la más rápida pero pequeña.
  • RAM (Principal): Es nuestra área de trabajo. Tarda entre 100-300 ciclos en responder, lo cual es muchísimo comparado con los registros.
  • Disco (SSD/HDD): El almacenamiento masivo. Es persistente (no se borra al apagar) pero es millones de veces más lento que los registros.
  • Dato curioso: Los registros son 100 millones de veces más rápidos que un disco duro tradicional.
